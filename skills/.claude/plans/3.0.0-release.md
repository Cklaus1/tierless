# Meridian 3.0.0 — Release Record

**Cut from:** `main` (2.4.1..main)
**Previous release:** 2.4.1
**Release owner:** chris.klaus1@gmail.com
**Date:** 2026-07-11
**Status:** BLOCKED — do not tag until Pre-Release Blockers (below) are cleared.

---

## 1. Version and justification

**3.0.0.** Major bump is forced by two specific changes, independently:

- **#817** removes the `created` response field from every endpoint (renamed to
  `created_at`, no alias). Any consumer reading `created` breaks on upgrade.
  A removed/renamed public response field is a breaking API change by definition.
- **#830** changes the behavior of POST /records for existing callers with
  unchanged requests (automatic detection/merging of duplicates). Same request,
  different outcome — and a destructive one (records the caller believes they
  created may be silently merged away). Breaking, and ungated.

Either alone requires 3.0.0. This is not a "big release therefore major" call;
#812/#819/#826 alone would be 2.5.0, and #814/#828 alone would be 2.4.2.

Note on #821 (rate limit 100→250): loosening a limit is not a semver break, but
it changes downstream load characteristics — flagged for consumer comms, not
for the version number.

## 3. Pre-release blockers (must change before this ships)

1. **#830 smart dedup must go behind a flag, default OFF.** As merged it
   silently rewrites POST /records semantics for all twelve consuming teams at
   the moment of deploy, with no opt-out and no per-team rollout. Merged-but-
   ungated behavior changes are not shippable. Required: add
   `smart_dedup_enabled` config flag defaulting to false; verify OFF path is
   byte-identical to 2.4.1 behavior (minus the other changes).
   The release does NOT wait for dedup to be *ready* — it waits for dedup to be
   *dormant*.
2. **#823 backfill must not drop the `name` column in this release.** Split and
   backfill `first_name`/`last_name`, dual-write to keep `name` populated, and
   defer the column drop to a later release (3.1.x or a standalone migration)
   after a soak period. As merged, the drop executes at deploy and destroys the
   rollback path (§5). Required: edit the migration to remove the `DROP COLUMN`
   step; file a follow-up ticket for the deferred drop with an explicit
   "no rollback past this point" callout.
3. **#817 — decide alias vs. hard cut, deliberately.** Preferred: re-add
   `created` as a deprecated alias in 3.0.0 (emitting both fields), remove in
   4.0.0. If the team chooses the hard cut, 3.0.0 stands, but consumer lead
   time (§6) becomes a hard gate, not a courtesy.

## 4. Flags: shipping OFF vs. flipping ON

**Shipping OFF (dormant in the binary):**
- `smart_dedup_enabled` = false (per blocker #1). No flip planned inside the
  3.0.0 rollout. Flipping is its own future rollout with its own stages, own
  comms, and per-team opt-in first.

**Flipping ON with the release (behavior live at deploy):**
- #812 param, #819 bulk endpoint, #826 trace header — additive, always-on, no
  flag needed.
- #817 rename — live at deploy (or dual-field if alias option chosen).
- #814, #828 fixes — live at deploy.
- #821 new rate-limit default — live at deploy.

**Rollout stages** (conservatism justified in §5):
1. **Stage 0 — pre-deploy:** additive schema migration only (add columns,
   backfill, dual-write; NO drop). Verify 2.4.1 code still runs green against
   migrated schema. This proves the rollback path before we need it.
2. **Stage 1 — canary (1 instance / ~5%):** hold 24h minimum. Watch: 4xx/5xx on
   POST /records and PATCH, consumer client errors mentioning `created`,
   pagination metrics, dual-write consistency.
3. **Stage 2 — 25%:** hold 24h. Same signals plus per-consumer-team error rates.
4. **Stage 3 — 100%.**
5. **Stage 4 (separate, weeks later):** drop `name` column after soak + explicit
   sign-off. Point of no return; announced as such.

Gate rule: any consumer-visible error spike attributable to #817 or the
migration halts promotion and triggers rollback assessment, not "fix forward."

## 6. Migration status

- **#823:** as merged: split + backfill + drop-at-deploy. **Must be amended**
  to split + backfill + dual-write, drop deferred (blocker #2). Until amended:
  migration status = NOT SAFE.
- After amendment: forward-compatible (2.4.1 code reads `name`, which dual-write
  keeps fresh) and reversible (drop hasn't happened).
- Backfill runtime on the production `records` table must be measured on a
  prod-sized copy before Stage 0; a long-running backfill inside the deploy
  window is its own outage risk. Run it as an online/batched backfill, not a
  deploy-blocking step.

## 5. Rollback path — honest assessment

**With the change set as merged, there is no rollback.** PR #823's backfill
drops the `name` column at deploy time; 2.4.1 application code reads `name`
directly. "Redeploy the previous version" would crash on first read. A schema
restore from backup means data loss for everything written since deploy. So as
merged, the only remediation for any 3.0.0 problem — including problems in
completely unrelated PRs like the dedup feature or the field rename — is
fix-forward under incident pressure. That is why blocker #2 is a blocker and
not a preference.

**After blocker #2 is fixed (dual-write, deferred drop):**
- Rollback = redeploy 2.4.1. Old code reads `name` (kept fresh by dual-write)
  and ignores the new columns. Verified explicitly in Stage 0.
- Residual asymmetries even then, to have eyes open about:
  - Rows written by 3.0.0 populate `first_name`/`last_name` authoritatively;
    dual-write must be the mechanism that keeps `name` consistent, and it must
    be tested with weird names (single token, multi-token, empty).
  - If dedup were ever ON and merged records, rollback does not un-merge them —
    another reason it ships OFF and flips only in a dedicated later rollout.
- **Implication for stage gates:** rollback is cheap only until Stage 4. Between
  now and the column drop, gates can rely on "roll back if in doubt." Stage 4
  removes that option permanently, which is why it is weeks out, separately
  announced, and separately signed off — never bundled into the 3.0.0 promotion
  sequence.

## 7. Consumer communications

Twelve teams pin `^2.4.0`, so **none of them auto-adopt 3.0.0** — semver ranges
protect them from the client-library breaks. But the *server* changes hit them
at our deploy regardless of what client version they pin. The comms plan must
distinguish these:

**T-2 weeks (minimum) — breaking-change notice, before any deploy:**
- #817: `created` is now `created_at` in all responses **server-side at our
  deploy time**, regardless of their pinned client version (unless the alias
  option is taken — then this becomes a deprecation notice with a 4.0 removal
  date). Ask each team to confirm whether they read `created`; track acks.
  Do not deploy until the teams that read `created` have either migrated or
  we've added the alias.
- 3.0.0 exists; `^2.4.0` will not pick it up; here is the migration guide and
  the upgrade path.

**T-1 week — release preview:**
- Full changelog (this table), canary schedule with dates, rollback posture.
- #814: the empty-last-page bug is fixed — if you special-cased zero-item
  pages as end-of-results, verify your pagination loop still terminates.
- #821: default rate limit rises 100 → 250 req/min; if you sized retry/backoff
  or downstream capacity around 100, expect up to 2.5x throughput from bursty
  callers.
- #826: new `X-Meridian-Trace` response header; include it in bug reports.
  Flag for anyone with strict response-header validation or header-size limits
  at their proxy.
- #830: smart dedup exists but is OFF; it will get its own opt-in rollout and
  its own announcement. No action now.

**T-0 — deploy day:** deploy start/finish per stage, status channel, named
on-call, one-line rollback statement ("rollback to 2.4.1 is available until
the deferred column drop, scheduled separately").

**Post-release:** confirmation at 100%, reminder of the Stage 4 column-drop
date and its no-rollback consequence, dedup opt-in program announcement.

## 8. Release-day traps (explicit)

1. **#823 is the deploy-time landmine.** The drop runs *when the migration
   runs*, not when traffic shifts. Even a canary deploy of one instance runs
   the migration once and breaks the schema for the 95% of instances still on
   2.4.1 — the "canary" instantly takes down the whole fleet. Blocker #2 fixes
   this; without it, there is no safe canary at all.
2. **#830 is silently live at deploy.** Nothing gates it; the moment 3.0.0
   serves traffic, every POST /records from every team goes through dedup and
   may merge their data destructively. It will look like data loss to consumers
   and won't show up as an error in our dashboards. Blocker #1.
3. **#817 breaks consumers at *our* deploy, not at *their* upgrade.** Easy to
   assume semver pinning protects them; it protects them from the client
   library, not from the server's response body changing under their current
   client.
4. **Version-number anchoring:** the additive PRs make this feel like "2.5.0
   with some cleanup." Tagging it 2.5.0 would push breaking changes into
   consumers' `^2.4.0` ranges via the client library — the worst possible
   failure mode for a pinned ecosystem.
5. **Backfill duration:** an unbenchmarked full-table backfill in the deploy
   path can turn deploy into an outage even if everything else is right.
