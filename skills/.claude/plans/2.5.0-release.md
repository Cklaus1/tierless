## Version Bump Justified By

**MAJOR bump to 3.0.0.**

Two independent, unaliased breaking changes ship in this set:

1. **PR #823 (data migration):** The deploy-time backfill drops the old `name` column. Application code in 2.4.1 reads `name` directly. Redeploying 2.4.1 against the migrated schema will crash at runtime -- the column no longer exists. This is a forward-incompatible schema change.

2. **PR #817 (response field rename):** The `created` response field is gone, replaced by `created_at`. Consumers deserializing the response by field name will break. No aliasing was implemented.

Either one alone would warrant a MAJOR bump. Together, they are unambiguous: this is not a MINOR release.

**Verdict: version 3.0.0.**

## Flags Shipping OFF

These changes must NOT ship enabled-for-everyone as merged:

### PR #830 -- Smart dedup (OFF, must be gated)

The dedup logic changes what POST /records does to incoming data even for existing callers. It merges records automatically, which is a silent data mutation. This shipped merged without any feature flag or configuration toggle.

**Required before release:**
- Wrap the dedup logic behind a feature flag (e.g., `smart_dedup_enabled`).
- Default the flag to `false` in this release.
- Ship the flag OFF, then flip through stages:
  - Stage 1: Internal testing (1-2 days) -- verify dedup logic against production traffic shadow.
  - Stage 2: Canary 5% of traffic (3-5 days) -- watch for false-positive merges, review merged record quality.
  - Stage 3: 25% traffic (3-5 days) -- confirm no regressions in downstream consumers.
  - Stage 4: 100% traffic (2-3 days) -- full rollout.
- Each stage watches: dedup merge rate, false-positive rate (records incorrectly merged), error rate on POST /records, and downstream consumer error rates.

### PR #817 -- Response field rename (OFF or aliased)

The field rename has no backward-compatible alias. Consumers reading `response.created` will get `undefined`.

**Required before release:**
- Add a response alias: include both `created_at` (new) and `created` (deprecated, same value) in all responses.
- Document the deprecation window in the changelog.
- Remove the alias in a future MAJOR release (target: 4.0.0).
- If aliasing is not feasible, this change must be held out of this release and rescheduled for the 3.0.0 cut.

### PR #823 -- Schema migration (deferred or gated)

The backfill script runs at deploy time and drops the `name` column. This is the most dangerous change in the set because it makes rolling back impossible without a database restore.

**Required before release:**
- **Do not drop the `name` column in this release.** Keep it as a computed/deprecated column or simply leave it in place.
- The split into `first_name` / `last_name` can proceed (adding columns is reversible).
- The backfill can run to populate the new columns.
- The column drop must be a separate, later release with its own MAJOR bump, after consumers have migrated off `name`.
- If the column drop is absolutely required now, the rollback path is "restore from database backup" -- which is hours of downtime, not a redeploy. This is unacceptable for a release-day decision. The stage gates would need to be 100% conservative: full production validation before any flip.

## Release-Day Traps

1. **PR #823 column drop is a rollback killer.** If the `name` column is dropped at deploy time, the previous version (2.4.1) cannot run against the new schema. This is the single highest-risk item in the entire change set. If the column drop ships, the team must be prepared for a database restore rollback -- which means the canary stage should be 100% of traffic validated before promotion, because there is no partial rollback.

2. **PR #830 silently mutates data.** Smart dedup merges records automatically. If this ships enabled, existing consumers will see their records disappear (merged into others) without any API-level signal. This is not a breaking API contract change -- it is a silent data behavior change. The blast radius is every POST /records call. Must be flagged OFF.

3. **PR #817 has no alias.** The field rename shipped without backward compatibility. Any consumer that accesses `response.created` will get `undefined`. This is a silent break -- no HTTP error, no schema violation, just missing data. Must be aliased or held.

4. **PR #821 rate-limiter bump (100 -> 250 req/min) is a double-edged sword.** It is not a breaking change per se, but it changes the effective capacity of the service. Consumers who were previously rate-limited at 100 req/min may suddenly get through more requests, which could stress downstream systems (databases, other services). This should be monitored closely even though it is "just a config change."

5. **PR #831 (README update) is the only safe item.** Everything else has operational impact. The README update is the only thing that could ship without any staging.

**Do not cut 2.5.0 from the current state of main.** The change set contains unaliased breaking changes and a forward-incompatible schema migration that make the release process unsafe.

**Option A (recommended): Cut 2.5.0 with only safe changes.**
- Cherry-pick #812, #814, #819, #821, #826, #828, #831 onto a release branch.
- Hold #817, #823, #830 for 3.0.0.

- Rollback: redeploy 2.4.1 works for everything.
- Timeline: can ship this week.

**Option B: Cut 3.0.0 with all changes, but fix the safety gaps.**
- Add alias for `created` -> `created_at` (#817).
- Add feature flag for smart dedup (#830), ship OFF.
- Defer column drop for `name` (#823) -- keep `name` column, backfill new columns only.
- Version: 3.0.0 (MAJOR -- field rename is still breaking even with alias, or hold it).
- Rollback: redeploy 2.4.1 works for everything except #830 (flag-off).
- Timeline: minimum 2 weeks for consumer migration window.

**Option C (not recommended): Cut 3.0.0 as-is from main.**
- This is what the current state of main produces. It is the least safe option.
- Rollback is broken by #823.
- Smart dedup (#830) is enabled for everyone with no flag.
- Field rename (#817) has no alias.
- This is a release-day incident waiting to happen.
