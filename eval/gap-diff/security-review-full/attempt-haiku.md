# Security Review: Flask Password-Reset Flow

## VULNERABILITY 1 — Email Enumeration (Information Disclosure)

**Severity:** Medium
**Location:** `/forgot` route, lines 13-14

The endpoint returns HTTP 404 with the message `"no account with that email"` when the email does not exist in the database. This allows an attacker to enumerate valid email addresses registered with the application.

**Exploit path:**
1. Attacker sends `POST /forgot` with `{"email": "admin@company.com"}` and observes a 200 response (account exists).
2. Attacker sends `POST /forgot` with `{"email": "nonexistent@company.com"}` and observes a 404 response (account does not exist).
3. Attacker iterates through a list of candidate emails to build a roster of valid accounts.

**Impact:** Enables targeted phishing, social engineering, and credential-stuffing campaigns against known accounts. Combined with the token generation weakness below, it narrows the attacker's attack surface to real users.

**Fix:** Return the same HTTP status code (200) and a generic response for both cases: `{"ok": true}` regardless of whether the email exists. Optionally add a rate limit to prevent bulk enumeration.

## VULNERABILITY 2 — Weak Token Generation (Cryptographic Weakness)

**Severity:** High
**Location:** `/forgot` route, line 15

```python
token = hashlib.md5((email + str(time.time())).encode()).hexdigest()
```

The reset token is derived from `hashlib.md5(email + timestamp)`. This has three compounding weaknesses:

**(a) MD5 is unsuitable for security tokens.** MD5 is cryptographically broken and collision-prone. While collision resistance is not the primary concern here (the attacker does not control the input), MD5's speed means it provides negligible computational cost to an attacker attempting brute-force or precomputation attacks.

**(b) The timestamp component has low entropy.** `time.time()` returns a Unix timestamp with second-level precision. An attacker can narrow the possible token space to a small window. If the attacker knows approximately when the token was generated (e.g., by triggering the request and observing the response time), the effective entropy is limited to the seconds within that window. Even a 60-second window means only 60 possible inputs to hash for a given email.

**(c) The email is predictable.** The attacker knows or can guess the victim's email. Combined with the low-entropy timestamp, the total input space to the hash function is extremely small: `len(email_guesses) x seconds_in_window`. For a known email and a 60-second window, this is at most ~60 hash computations — trivially brute-forced.

**Exploit path:**
1. Attacker triggers a password-reset request for a known email (or enumerates one via Vulnerability 1).
2. Attacker records the response timestamp.
3. Attacker iterates over a window of timestamps (e.g., +/- 60 seconds around the observed time), computes `md5(email + str(t))` for each, and checks if the resulting token matches any token observed in the reset email (if the attacker has partial access to mail or can observe the token via a side channel).
4. Even without observing the token, if the attacker can guess the token format or if the token is leaked through logs, error messages, or any other channel, the weak generation makes it trivially reversible.

**Impact:** An attacker can forge or brute-force valid reset tokens, gaining full access to the victim's account and the ability to reset their password.

**Fix:** Use `secrets.token_urlsafe(32)` or `secrets.token_hex(32)` to generate cryptographically random tokens. Store the token and its expiration time in the database (not an in-memory dict). Never derive tokens from predictable inputs.

## VULNERABILITY 3 — In-Memory Token Store (No Persistence, No Expiration)

**Severity:** High
**Location:** Lines 7, 16, 24

```python
reset_tokens = {}   # token -> email
```

**(a) No persistence.** The `reset_tokens` dictionary is stored in process memory. If the Flask process restarts (crash, deployment, scaling), all reset tokens are lost. Legitimate users cannot complete password resets after a restart. In a multi-worker or containerized deployment (e.g., multiple Gunicorn workers, Kubernetes pods), a token generated by one worker may not be available to the worker that handles the reset request, causing legitimate resets to fail.

**(b) No expiration.** Tokens never expire. A leaked or intercepted token remains valid indefinitely. There is no TTL, no max-use count, and no mechanism to revoke a token after it has been used.

**(c) No single-use enforcement.** The same token can be used repeatedly to reset the password. An attacker who obtains a token (even an old one) can call `/reset` multiple times.

**Exploit path:**
1. Attacker obtains a valid reset token through any means (mail interception, log leakage, brute-force due to weak generation).
2. Attacker calls `POST /reset` with the token and a new password.
3. The victim's password is changed.
4. The attacker can repeat step 3 indefinitely (no single-use enforcement).
5. Even if the victim changes their password back, the attacker can reset it again with the same token.

**Impact:** Permanent account takeover with no time limit on the attack window.

**Fix:** Store tokens in a persistent store (database, Redis) with:
- A TTL (e.g., 1 hour).
- A `used` flag that is set to `true` after successful use.
- A database entry that is deleted after expiration.

## VULNERABILITY 4 — No Rate Limiting on /forgot

**Severity:** Medium
**Location:** `/forgot` route

There is no rate limiting on the `/forgot` endpoint. An attacker can send an unlimited number of requests to trigger password-reset emails for any email address.

**Exploit path:**
1. Attacker sends thousands of `POST /forgot` requests for different email addresses.
2. This causes a denial of service through email-sending costs (billable per email) and inbox flooding for victims.
3. Combined with Vulnerability 1 (email enumeration), the attacker can efficiently enumerate emails while simultaneously spamming victims with reset emails (which may cause the victim to ignore a legitimate reset email later).

**Impact:** DoS via email flooding, increased infrastructure costs, and degraded user experience.

**Fix:** Implement rate limiting (e.g., using Flask-Limiter) at the IP level and/or email level. Allow at most 1-2 requests per email per hour.

## VULNERABILITY 5 — No Rate Limiting on /reset

**Severity:** High
**Location:** `/reset` route

There is no rate limiting on the `/reset` endpoint. Combined with the weak token generation (Vulnerability 2), an attacker can brute-force tokens at arbitrary speed.

**Exploit path:**
1. Attacker generates candidate tokens (using the weak MD5+timestamp method).
2. Attacker sends rapid-fire `POST /reset` requests with each candidate token.
3. Without rate limiting, the attacker can try thousands of tokens per second.
4. Given the low entropy of the token space (Vulnerability 2), the attacker will find a valid token quickly.

**Impact:** Rapid automated account takeover.

**Fix:** Implement strict rate limiting on `/reset` (e.g., 5 attempts per IP per minute). Additionally, use strong random tokens (Vulnerability 2 fix) so brute-force is computationally infeasible even without rate limiting.

## VULNERABILITY 6 — Password Stored in Plaintext

**Severity:** Critical
**Location:** `/reset` route, line 27

```python
db.users.update_one({'email': email}, {'$set': {'password': new_password}})
```

The new password is stored directly in the database without hashing. This means:

**(a) If the database is compromised, all user passwords are exposed in plaintext.** There is no bcrypt, scrypt, Argon2, or PBKDF2 hashing.

**(b) The application likely stores all passwords in plaintext, not just newly reset ones.** If the initial password storage also lacks hashing, the entire user base is at risk.

**(c) No password strength validation.** The endpoint accepts any `new_password` value without checking minimum length, complexity, or against known breached passwords. An attacker can set a very weak password.

**Exploit path:**
1. Attacker gains read access to the database (via SQL/NoSQL injection, misconfigured access controls, backup exposure, or insider threat).
2. Attacker extracts all user passwords in plaintext.
3. Attacker uses these passwords for credential stuffing against other services (users frequently reuse passwords).

**Impact:** Complete credential compromise of all user accounts, with downstream effects on other platforms where users reuse passwords.

**Fix:** Hash passwords using a proper algorithm (bcrypt, Argon2id, or scrypt) before storing. Validate password strength before accepting it. Use `werkzeug.security.generate_password_hash()` and `check_password_hash()` if staying within the Flask ecosystem.

## VULNERABILITY 7 — No CSRF Protection

**Severity:** High
**Location:** Both `/forgot` and `/reset` routes

Both endpoints accept state-changing requests via POST but have no CSRF protection. There is no mention of a CSRF token check, and the code does not verify the `Origin` or `Referer` header.

**Exploit path for `/reset`:**
1. Attacker creates a malicious webpage with a hidden form:
   ```html
   <form action="https://app.com/reset" method="POST">
     <input name="token" value="attacker_obtained_token">
     <input name="new_password" value="attacker_controlled_password">
     <input type="submit">
   </form>
   <script>window.onload = function() { document.forms[0].submit(); }</script>
   ```
2. Victim (who has a valid reset token in their email) visits the attacker's page while logged into `app.com`.
3. The browser automatically includes any cookies for `app.com` with the request.
4. The victim's password is changed to the attacker's password without their knowledge.

**Note:** If the application uses cookie-based session authentication, this exploit works even without the victim being actively logged in at the moment, as long as a valid session cookie exists.

**Impact:** Account takeover via CSRF, without any interaction from the victim beyond visiting a malicious page.

**Fix:** Implement CSRF tokens using Flask-WTF or a similar library. Verify the `Origin` and `Referer` headers as an additional defense-in-depth measure. Consider using the SameSite cookie attribute on session cookies.

## VULNERABILITY 8 — No Input Validation / Sanitization

**Severity:** Medium
**Location:** Both `/forgot` and `/reset` routes

**(a) No validation of email format.** The `/forgot` endpoint passes `request.json['email']` directly to the database query without validating that it is a well-formed email address. An attacker could send malformed input that causes unexpected behavior.

**(b) No validation of `new_password`.** The `/reset` endpoint accepts any value for `new_password` without checking for minimum length, maximum length, or dangerous characters. While MongoDB handles the storage, an extremely long password could cause database issues or DoS.

**(c) No validation of `token`.** The `/reset` endpoint accepts any value for `token` without length or format validation.

**(d) No error handling for missing keys.** If `request.json` does not contain the expected keys (`email`, `token`, `new_password`), the code will raise a `KeyError`, which could result in a 500 Internal Server Error with a stack trace exposed to the attacker.

**Exploit path:**
1. Attacker sends `POST /forgot` with `{"not_email": "test"}` (missing the `email` key).
2. The code raises `KeyError: 'email'`, potentially returning a 500 error with a stack trace.
3. The stack trace may reveal internal details: file paths, database structure, library versions, etc.

**Impact:** Information disclosure via stack traces, potential DoS via malformed input, and bypass of input validation controls.

**Fix:** Validate all inputs before processing. Use a schema validation library (e.g., marshmallow, pydantic, or Flask-WTF forms). Wrap endpoint logic in try/except blocks that return generic error messages without exposing internal details.

## VULNERABILITY 9 — Timing Side-Channel on Token Lookup

**Severity:** Low
**Location:** `/reset` route, line 24

```python
email = reset_tokens.get(token)
```

While `dict.get()` in Python is O(1) and does not have a timing side-channel, the overall flow may still be vulnerable depending on the database lookup in `/forgot`:

```python
user = db.users.find_one({'email': email})
```

If `db.users.find_one()` performs a string comparison that short-circuits on the first differing character (which is common in many database drivers), an attacker can measure response times to determine how many leading characters of an email address are correct. This is a classic timing oracle.

**Exploit path:**
1. Attacker sends many `POST /forgot` requests with emails like `a@x.com`, `ab@x.com`, `abc@x.com`, etc.
2. Attacker measures response times for each.
3. The email that takes slightly longer has one more correct leading character.
4. Attacker iteratively builds the email address character by character.

**Impact:** Slow email enumeration, which is less precise than the HTTP-status-based enumeration in Vulnerability 1 but still useful.

**Fix:** Use constant-time comparison for any string comparisons derived from user input. For MongoDB, ensure the driver does not short-circuit. More importantly, fix Vulnerability 1 (email enumeration via HTTP status) which is the more practical and exploitable vector.

## VULNERABILITY 10 — Token Leaked in URL (Log/Referrer Exposure)

**Severity:** Medium
**Location:** `/forgot` route, line 17

```python
send_email(email, f"Reset here: https://app.com/reset?token={token}")
```

The reset token is passed as a query parameter in the URL. This means:

**(a) The token appears in email body logs** of mail servers and email clients.
**(b) The token appears in browser history** when the user clicks the link.
**(c) The token may be leaked via the `Referer` header** if the reset page redirects to a third-party site or if the user clicks a link on the reset page.
**(d) The token may appear in server access logs** on both the client side (if the user's browser logs referer) and the server side (the reset endpoint receives the token in the URL before the POST body).

**Exploit path:**
1. Attacker gains access to server access logs (via misconfiguration, log aggregation vulnerability, or insider threat).
2. Attacker extracts reset tokens from the URL query parameters.
3. Attacker uses the token to reset the victim's password.

**Impact:** Token leakage through multiple channels, enabling account takeover without needing to intercept the email itself.

**Fix:** Pass the token in the POST body (as the code already does for `/reset`), not in the URL. Send an email with a link to a GET endpoint (e.g., `/reset-page?token=...`) that displays a form, and have the form submit the token via POST to `/reset`. Alternatively, use a POST-only endpoint and include the token in a hidden form field.

## VULNERABILITY 11 — No Authentication Required on /reset

**Severity:** High
**Location:** `/reset` route

The `/reset` endpoint does not require any authentication. Anyone with a valid token can reset the password. While this is partially by design (the token is the credential), there is no additional binding to the user's session, IP address, or device fingerprint.

**Exploit path:**
1. Attacker obtains a reset token (via any means: brute-force, mail interception, log leakage).
2. Attacker resets the password from any IP address, any device, any location.
3. There is no check that the reset request comes from the same network or device that received the email.

**Impact:** No contextual validation of the reset request. Any token holder can perform the reset from anywhere.

**Fix:** While binding to IP or session is not a complete defense (tokens are the credential), consider adding:
- A user agent check (store the user agent when the token is generated, verify it on reset).
- A rate limit per token (e.g., max 5 attempts per token).
- Notification to the victim when a password reset occurs.
